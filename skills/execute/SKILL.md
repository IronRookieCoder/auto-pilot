# TDD 驱动的逐里程碑执行

按照 `.workflow/plan.md` 中的里程碑序列，以严格的 TDD 循环逐个执行。每个里程碑遵循"写测试 → 实现 → 验证"的节奏。

## 上下文隔离策略

**关键机制**：每个里程碑通过子代理（Agent 工具）执行，完成后子代理上下文自动释放。execute 作为编排者保持精简上下文。

```
execute（编排者）
  ├─ 读取 plan.md → 定位下一个里程碑
  ├─ 生成子代理 → 执行 M0 的 TDD 循环 → 返回摘要 → M0 上下文释放
  ├─ 更新 plan.md（折叠 M0）+ documentation.md + changelog.md
  ├─ 生成子代理 → 执行 M1 的 TDD 循环 → 返回摘要 → M1 上下文释放
  ├─ 更新 plan.md（折叠 M1）+ documentation.md + changelog.md
  └─ ...直到全部完成
```

**为什么**：里程碑 M15 执行时，M0-M14 的代码 diff、测试输出、错误信息已不在上下文中，AI 注意力完全集中在当前里程碑。

## 前置检查

1. 确认 `.workflow/` 目录存在且包含所有 5 个文件（spec/plan/implement/documentation/changelog）
2. 确认 `plan.md` 中有里程碑定义（不是空骨架）
3. 读取 `implement.md` 加载执行规则和验证命令
4. 读取 `spec.md` 了解项目目标和约束
5. 读取 `documentation.md` 了解当前状态和活跃问题

**注意力保护读取策略**：
- 每个里程碑开始时只读 plan.md 的"当前及待办里程碑"区（跳过"已完成里程碑"折叠区）
- documentation.md 只看当前状态、活跃问题、待澄清事项（该文件控制在 100 行以内）
- **不读 changelog.md**，除非遇到需要排查历史上下文的问题

## 编排者流程

execute 自身作为编排者，执行以下循环：

```
1. 读取 plan.md → 找到下一个 🔲 待开始 的里程碑
2. 将里程碑状态更新为 🔶 进行中
3. 用 Agent 工具生成子代理，传入里程碑执行 prompt（见下文）
4. 子代理返回执行结果（成功/失败/阻塞 + 摘要）
5. 根据结果：
   ├─ 成功 → 折叠里程碑到 plan.md + 更新 documentation.md + 追加 changelog.md
   ├─ 失败/阻塞 → 记录到 documentation.md，报告用户
6. 回到步骤 1
```

### 子代理 prompt 模板

为每个里程碑生成子代理时，传入以下信息：

```
你正在执行项目 "{项目名称}" 的里程碑 M{N}: {名称}。

## 执行规则
{implement.md 的完整内容}

## 里程碑定义
{从 plan.md 提取的当前里程碑完整内容}

## 项目上下文
- 技术栈：{从 spec.md 提取}
- 活跃问题：{从 documentation.md 提取}
- 验证命令：{从 implement.md 提取}

## 你的任务
严格按照 TDD 循环执行此里程碑：
1. 编写测试用例（基于测试设计）
2. 运行测试，确认 RED
3. 编写实现代码
4. 运行验证（lint + typecheck + test + build）
5. 验证失败时只修改实现代码，禁止修改测试

完成后返回：
- 状态：成功 / 失败 / 阻塞
- 测试结果：{通过数}/{总数}
- 新增/修改的文件列表
- 关键决策（如有）
- 失败原因（如有）
```

**子代理不直接写 plan.md / documentation.md / changelog.md**，只负责编写代码和测试、运行验证。文档更新由编排者根据子代理返回的结果完成。

## 里程碑 TDD 循环（子代理内部）

对于每个里程碑，子代理严格执行以下 TDD 循环：

```
┌─────────────────────────────────────────────────────────┐
│                    里程碑 TDD 循环                        │
│                                                          │
│  1. 读取里程碑定义（范围、测试设计、验收标准）              │
│             ↓                                            │
│  2. 编写测试用例（基于测试设计）                           │
│             ↓                                            │
│  3. 运行测试 → 确认全部 RED（失败）                       │
│             ↓                                            │
│  4. 编写实现代码                                          │
│             ↓                                            │
│  5. 运行验证（lint + typecheck + test + build）           │
│         ├─ 全部通过 → 返回成功                            │
│         └─ 失败 → 修改实现代码（禁止改测试）→ 回到步骤 5   │
│                  （最多 3 轮，仍失败则返回阻塞）           │
└─────────────────────────────────────────────────────────┘
```

## 详细步骤

> 步骤 1、5、6、7 由编排者执行。步骤 2、3、4 由子代理执行（通过 prompt 传入规则）。

### 步骤 1：定位下一个里程碑 [编排者]

从 `plan.md` 中找到第一个状态为"🔲 待开始"的里程碑。检查：
- 该里程碑的依赖是否全部完成
- 如果有未完成的依赖，跳到可执行的里程碑或报告阻塞

将该里程碑状态更新为"🔶 进行中"。

### 步骤 2：编写测试用例（RED 阶段）[子代理]

根据里程碑中的"测试设计"部分：

1. **创建测试文件**（如果不存在）
2. **编写测试用例**，覆盖里程碑定义的所有测试项
3. 测试用例应该：
   - 测试预期行为，而非实现细节
   - 包含正常路径和边界情况
   - 有清晰的测试描述
4. **运行测试，确认全部失败（RED）**
   - 如果部分测试已经通过，说明功能可能已部分实现，记录并继续

### 步骤 3：实现代码（GREEN 阶段）[子代理]

编写最小必要的实现代码使测试通过：

1. **只实现里程碑范围内的功能** - 不多不少
2. 遵循 `implement.md` 中的项目约定
3. 参考里程碑中的"关键文件"列表
4. 重要决策在返回结果中报告，由编排者统一记录

### 步骤 4：验证（VERIFY 阶段）[子代理]

运行完整验证序列：

```bash
Lint → TypeCheck → Test → Build
```

**验证规则（TDD 铁律）：**

- ✅ 可以修改实现代码来通过测试
- ✅ 可以修改实现代码来通过 lint/typecheck
- ✅ 可以添加新的实现文件
- ❌ 不能修改测试文件中的断言或预期值
- ❌ 不能删除或跳过测试用例
- ❌ 不能放宽 lint 规则或 type 检查
- ❌ 不能在里程碑范围外添加功能

**修复循环：**
- 验证失败时，分析错误 → 修改实现代码 → 重新验证
- 单个验证步骤最多 3 轮修复
- 3 轮后仍失败：返回"阻塞"状态和失败详情，由编排者记录到 documentation.md 并报告用户

### 步骤 5：标记完成并折叠 [编排者]

验证全部通过后：

1. 在 `plan.md` 中将里程碑完整内容折叠为单行摘要，移入"已完成里程碑"区：
   ```
   ✅ M{N}: {名称} | 测试 {通过}/{总数} | 完成于 {日期} | 决策: {关键决策或"无"}
   ```
2. 从"当前及待办里程碑"区删除该里程碑的完整内容
3. 记录完成时间

### 步骤 6：更新文档 [编排者]

**changelog.md** — 追加详细执行日志：

```markdown
### {日期} - M{N}: {里程碑名称} ✅

- **事件**：里程碑完成
- **实现摘要**：{做了什么}
- **新增/修改文件**：
  - `path/to/file` - {说明}
- **测试结果**：{N 通过, 0 失败}
- **决策记录**：{重要决策，如有}
- **耗时**：{从开始到完成}
```

**documentation.md** — 更新仪表盘：
- 更新进度概览表的数值（完成数、测试指标）
- 更新"当前里程碑"为下一个待执行的里程碑
- 更新"最近活动"（只保留最近 3 条，旧条目已在 changelog.md）
- 已解决的问题从"活跃问题"中移除（已在 changelog.md 归档）

### 步骤 7：继续下一个里程碑 [编排者]

自动继续下一个里程碑，回到步骤 1。

**停止条件：**
- 所有里程碑完成 → 输出完成摘要
- 遇到阻塞的里程碑 → 报告并等待用户干预
- 用户中断 → 保存当前进度

## 中断恢复

如果执行被中断（上下文切换、用户暂停等），重新调用时：
1. 读取 `plan.md` 找到最近的"🔶 进行中"或第一个"🔲 待开始"里程碑
2. 读取 `documentation.md` 了解上次停在哪里
3. 从断点继续执行

## 完成摘要

所有里程碑完成后，输出最终报告：

```
## 执行完成报告

📊 完成度：{N}/{N} (100%)
⏱ 总耗时：{开始到结束}

### 里程碑回顾
| # | 名称 | 测试 | 决策 |
|---|------|------|------|
| M0 | ... | 3/3 | - |
| M1 | ... | 5/5 | 选择了 X 方案 |

### 最终验证
{完整验证结果}

### 下一步建议
- 运行 /auto-pilot:verify 做最终全量检查
- 审阅 .workflow/documentation.md 中的决策记录
- 检查是否有待澄清事项需要处理
```

## 重要约束

- **严格 TDD 循环**：先写测试，后写实现，不可颠倒
- **禁止修改测试以通过验证**：这是贯穿整个工作流的铁律
- **一次一个里程碑**：不并行执行多个里程碑
- **范围纪律**：只做里程碑定义的内容，不"顺便"改其他东西
- **完整记录**：每个里程碑的执行过程都有迹可查
- **验证不可跳过**：每个里程碑必须通过完整验证才能标记完成
