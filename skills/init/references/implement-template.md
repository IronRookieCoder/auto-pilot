# {{PROJECT_NAME}} - 执行指令

> 本文件是 AI 编码代理的"操作规程"，定义执行阶段的规则和约束。
> 代理在每个里程碑执行前必须读取本文件，确保行为一致。

## 核心执行原则

### 1. 范围纪律
- **严禁范围蔓延**：只做 plan.md 中定义的里程碑内容，不添加"顺便改进"
- **有疑问先记录**：遇到规格歧义时，记录到 documentation.md 的"待澄清"区，不自行决定
- **一次一个里程碑**：完成当前里程碑并通过验证后，才能开始下一个

### 2. 验证纪律（TDD 强约束）
- **测试先行**：每个里程碑开始时，先编写/确认测试用例，再编写实现代码
- **禁止修改测试用例以通过验证**：测试用例一旦确定，只能修改实现代码来满足测试，绝不允许反过来修改测试来适配实现
- **验证失败立即修复**：不跳过，不延后，当前里程碑的验证必须全部通过才能继续
- **验证命令必须执行**：不能假设"应该能通过"，必须实际运行验证命令

### 3. 记录纪律
- **每个里程碑完成后更新 plan.md**：标记完成状态、记录决策
- **重要变更记录到 documentation.md**：包括架构决策、API 变更、问题修复
- **失败也要记录**：验证失败的原因和修复过程都是有价值的信息

### 4. 文档管理纪律（注意力保护）
- **plan.md 里程碑折叠**：里程碑完成后，将其完整内容折叠为单行摘要，移入"已完成里程碑"区
- **documentation.md 保持精简**：只保留当前状态、最近 3 条活动、活跃问题；已解决的问题和旧日志移入 changelog.md
- **changelog.md 是归档**：详细历史记录的存放处，日常执行不读取此文件
- **读取策略**：每个里程碑开始时，读取 spec.md + implement.md + plan.md（当前里程碑区域）+ documentation.md。不读 changelog.md 除非需要排查历史问题

### 5. 上下文隔离纪律
- **每个里程碑用子代理执行**：通过 Agent 工具为每个里程碑生成独立子代理，完成后上下文自动释放
- **编排者不写代码**：编排者只负责调度里程碑、更新工作流文件；代码编写和测试由子代理完成
- **文件是唯一的状态传递机制**：里程碑之间通过 .workflow/ 文件传递状态，不依赖上下文中的对话历史

## 执行循环

对每个里程碑，执行以下循环：

```
读取里程碑定义 → 编写测试用例 → 运行测试确认 RED（全部失败）→ 实现代码 → 运行验证 →
  ├─ 通过 → 更新 plan.md（标记完成）→ 更新 documentation.md → 下一个
  └─ 失败 → 分析原因 → 修复实现代码（禁止改测试）→ 重新验证 → ...
```
