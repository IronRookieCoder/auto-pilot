# 全流程自动编排

一键启动完整的 TDD 驱动编码工作流：自动检测状态 → init → plan → execute → 完成。支持中断后重新调用从断点继续。

## 状态检测与恢复

启动时自动检测 `.workflow/` 的当前状态，决定从哪个阶段开始：

```
检查 .workflow/ 目录
  ├─ 不存在 → 从 init 开始
  ├─ 存在但 spec.md 是空模板 → 提示用户填写 spec
  ├─ spec.md 已填写，plan.md 是空骨架 → 展示 spec.md 等待用户确认，确认后从 plan 开始
  ├─ plan.md 有里程碑，全部未开始 → 从 execute 开始
  ├─ plan.md 有进行中的里程碑 → 从断点继续 execute
  └─ plan.md 所有里程碑已完成 → 输出完成报告
```

## 执行流程

### 上下文隔离

run 作为顶层编排者，每个阶段通过子代理执行，阶段完成后上下文自动释放：

```
run（编排者，只做状态检测和阶段调度）
  ├─ 子代理: init → 完成, 上下文释放
  ├─ ⏸️ spec.md 用户确认（唯一人工门禁）
  ├─ 子代理: plan → 完成, 上下文释放
  ├─ execute（直接执行，内部再按里程碑生成子代理）
  └─ 子代理: verify（最终全量检查）→ 完成, 上下文释放
```

init 和 plan 产出全部写入 .workflow/ 文件，下一阶段从文件读取，不依赖上下文传递。

### 阶段 1：初始化（init）

如果 `.workflow/` 不存在：

1. 从用户输入中提取项目需求描述（用户调用 run 时应提供需求）
2. 执行 init 技能的完整流程
3. **使用用户提供的需求直接填充 spec.md**

**如果用户没有提供需求描述**：询问用户提供需求，不继续执行。

### 阶段 1.5：spec.md 确认（必需门禁）

init 完成后，**必须暂停等待用户确认 spec.md**，不得自动跳过：

1. 向用户展示 spec.md 的完整内容或关键摘要
2. 明确提示用户审阅以下重点：
   - 项目目标和范围是否准确
   - 约束条件是否完整
   - 验收标准是否可测试（能用自动化测试或命令判定通过/失败）
   - 范围边界是否清晰（哪些不做）
3. **等待用户明确确认**（如"确认""没问题""OK""继续"等肯定回复）
4. 如果用户提出修改意见：修改 spec.md → 重新展示 → 再次等待确认
5. **只有收到用户明确确认后，才能进入阶段 2**

```
⏸️ spec.md 审阅确认
请审阅以上规格说明，重点检查：
  ✅ 项目目标和范围是否准确？
  ✅ 约束条件是否完整？
  ✅ 验收标准是否可测试？
  ✅ 范围边界（不做什么）是否清晰？

👉 确认无误请回复"确认"，如需修改请说明修改内容。
```

### 阶段 2：规划（plan）

1. 执行 plan 技能的完整流程
2. 输出里程碑概览
3. **全自动模式：不暂停等待用户审阅**，直接进入执行
4. 用户如需审阅可中断后手动操作

### 阶段 3：执行（execute）

1. 执行 execute 技能的完整流程
2. 逐里程碑 TDD 循环：写测试 → 实现 → 验证
3. 每个里程碑完成后自动更新文档
4. 所有里程碑完成后进入收尾

### 阶段 4：收尾

1. 运行最终全量验证（verify 技能的完整流程）
2. 生成完成报告
3. 更新 documentation.md

## 全自动模式的约束

- **spec.md 确认是唯一的人工门禁**：init 完成后必须暂停等用户确认 spec.md，这是全流程中唯一需要人工介入的暂停点
- **确认后不再暂停**：plan → execute 连续执行
- **不在里程碑间暂停**：一个完成后自动开始下一个
- **但验证失败时必须修复**：不跳过不延后
- **TDD 铁律始终生效**：全自动不意味着降低质量标准

## 中断恢复

工作流可以在任何时候被中断，重新调用 `/auto-pilot:run` 时：

1. 读取 `.workflow/` 所有文件
2. 通过状态检测确定当前位置
3. 从断点继续执行
4. 不会重复已完成的里程碑

## 输出

### 启动时
```
🚀 工作流启动
📍 检测状态：{当前阶段}
📋 项目：{项目名称}
🎯 目标：{目标摘要}
```

### 每个阶段完成时
```
✅ {阶段名} 完成
   {关键信息}
→ 进入下一阶段：{阶段名}
```

### 全部完成时
```
🎉 工作流完成

📊 最终统计
- 里程碑：{N}/{N} 完成
- 测试：{N} 通过
- 验证：全部通过

📁 项目文件
- .workflow/spec.md           - 项目规格（冻结）
- .workflow/plan.md           - 里程碑计划（活跃+折叠）
- .workflow/implement.md      - 执行规则
- .workflow/documentation.md  - 状态仪表盘（≤100 行）
- .workflow/changelog.md      - 完整历史归档
```

## 重要约束

- **需求描述是必需的**：首次运行时用户必须提供需求，否则询问
- **不降低标准**：全自动不等于跳过验证或放松 TDD
- **幂等性**：重复调用从断点继续，不会破坏已有工作
- **记录完整**：全流程的每一步都有记录在 documentation.md 中
