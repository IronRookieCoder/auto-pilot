# 查看工作流状态

读取 `.workflow/` 目录中的记忆文件，生成项目进度摘要。

## 前置检查

- 确认 `.workflow/` 目录存在
- 至少 `plan.md` 存在（否则提示先运行 init 和 plan）

## 执行步骤

### 1. 解析里程碑状态

从 `.workflow/plan.md` 提取所有里程碑，统计：

| 状态 | 标记 | 说明 |
|------|------|------|
| 已完成 | ✅ | 已通过验证 |
| 进行中 | 🔶 | 当前正在执行 |
| 待开始 | 🔲 | 尚未开始 |

### 2. 生成进度摘要

```
## 项目进度 - {项目名称}

📊 总体进度：{已完成}/{总数} ({百分比}%)
████████░░░░ {进度条}

### 里程碑状态
| # | 名称 | 状态 | 测试 |
|---|------|------|------|
| M0 | 测试基础设施 | ✅ | 3/3 通过 |
| M1 | 核心模块 | 🔶 | 2/5 通过 |
| M2 | API 层 | 🔲 | - |
| ... | | | |

### 当前里程碑详情
- **名称**：M1 - 核心模块
- **进度**：测试 2/5 通过
- **剩余工作**：{未完成的验收标准}
- **阻塞项**：{如有}

### 最近活动
- {来自 documentation.md 的最近 3 条活动}

### 已知问题
- {来自 documentation.md 的未解决问题}

### 下一步
- {建议的下一步操作}
```

### 3. 更新 documentation.md

在 documentation.md 的进度概览表中更新最新数据，更新"最近活动"列表。

### 4. 检查测试健康度

如果项目有测试：
- 统计测试用例总数、通过数、失败数、跳过数
- 如有跳过的测试，标记为需要关注的风险项
- 更新 documentation.md 中的测试指标

### 5. 查阅历史（按需）

如果需要详细的历史信息（如排查某个决策的上下文），从 `changelog.md` 中查阅。日常 status 调用不读取 changelog.md。

## 输出格式

直接在终端输出格式化的进度摘要，同时更新 documentation.md。

## 独立调用模式

如果 `.workflow/plan.md` 不存在但项目有测试，仅输出测试健康度摘要。

## 重要约束

- **只读操作为主**：status 不修改 spec.md、plan.md、implement.md
- **只更新 documentation.md**：仅更新进度概览表和追加状态查看日志
- **信息要准确**：从文件实际内容解析，不猜测状态
